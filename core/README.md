# 🛡️ :core Module (도메인 및 애플리케이션 비즈니스 핵심)

## 📌 개요
본 모듈은 시스템의 심장이자 그 어떤 외부 프레임워크(Spring 등)나 데이터베이스 기술에도 오염되지 않는 **순수 무균실(Pure Domain & Application Core)** 영역입니다.

---

## 🧪 비즈니스 무결성 검증 정책 (Unit Test Specification)
본 모듈의 핵심 유즈케이스(`RecordTransactionService`)는 시스템의 안전성을 증명하기 위해 스프링 부트를 띄우지 않는 **순수 코틀린 단위 테스트(`RecordTransactionServiceTest`)**로 3초 이내에 무결성을 완벽히 검증합니다.

기여자들은 코드를 수정하거나 기능을 추가할 때, 아래 명시된 **3대 핵심 방어선** 테스트 규격을 반드시 준수해야 합니다.

### 1. 쓰레기 데이터 진입 원천 차단 (입구 컷 검증)
- **의도:** 가계부의 원천 데이터가 오염되는 것을 막기 위해 비즈니스 로직(Service) 전단계에서 쓰레기 데이터를 걸러내야 합니다.
- **검증 방식:** 결제 금액이 `0원` 이하인 유입에 대해, 배달통 객체인 `RecordTransactionCommand` 생성 시점에 즉각 `IllegalArgumentException`을 던지며 프로세스가 컨트롤러 단에서 폭사하는지 행위를 검증합니다.

### 2. 날것 데이터의 표준화 금융 규격화 (Money VO 제련 검증)
- **의도:** 외부에서 들어오는 소수점 오차 위험이 있는 `Double` 데이터를 안전한 금융 연산용 원자재로 변환해야 합니다.
- **검증 방식:** 중앙 지휘자(`RecordTransactionService`)가 데이터를 받는 즉시, 내부적으로 `BigDecimal` 기반의 `Money` 값 객체(Value Object)로 오차 없이 완벽하게 변환해 내는지 값의 상태를 검증합니다.

### 3. 비즈니스 정체성 부여 및 영속화 위임 (엔티티 조립 & 포트 호출 검증)
- **의도:** 파편화된 데이터에 가계부만의 고유 정체성을 부여하고, 핵심 로직이 끝난 후 데이터를 영속성 계층으로 안전하게 토스하는지 확인합니다.
- **검증 방식:**
    1. 가계부 고유 식별자가 도메인 규칙에 따라 **`TX-`로 시작하는 고유 UUID**로 자동 발급되어 `Transaction` 엔티티에 채워지는지 검증합니다.
    2. 조립이 끝난 엔티티가 아웃바운드 포트인 `SaveTransactionPort.save()`를 **정확히 딱 1번 호출**하여 외부로 안전하게 위임되는지 MockK를 통해 행위를 검증합니다.

---

## 🚨 기여자 유의사항
- `:core` 모듈 내의 코드를 수정할 경우, 반드시 `./gradlew :core:test` 명령어를 수행하여 위 3가지 비즈니스 방어선이 깨지지 않는지 확인하십시오.
- 새로운 유즈케이스를 추가할 때도 위와 같은 3단계 검증(입구 컷, 도메인 조립, 포트 위임) 흐름을 따르는 테스트 코드를 동반해야 PR이 승인됩니다.
