# ⚙️ 애플리케이션 서비스 및 비즈니스 오케스트레이션 정책

이 문서는 Money-Log 프로젝트의 애플리케이션 계층 중앙에서 비즈니스 로직을 지휘하는 **애플리케이션 서비스(Application Service / Use Case Implementation)**의 설계 철학과 구현 규칙을 명시합니다. 외부 개발자가 프로젝트에 참여했을 때 이 가이드를 통해 아키텍처의 위반 없이 신속하게 비즈니스 로직을 확장할 수 있도록 돕습니다.

---

## 1. 아키텍처 내 위치 및 역할 (Context & Role)

헥사고날 아키텍처에서 애플리케이션 서비스는 **육각형의 심장(도메인) 바로 바깥쪽을 감싸는 핏줄이자 오케스트라의 지휘자**입니다.

- **인바운드 포트의 계약 구현:** 외부 세계(웹, 앱, 메시지 큐 등)가 시스템에 요청을 보낼 때 통과하는 정문 인터페이스(`UseCase`)의 실제 구현체입니다.
- **아웃바운드 포트의 호출 제어:** 비즈니스 연산이 끝난 결과를 외부 세계(데이터베이스, 외부 시스템 등)로 내보내기 위해 뒷문 인터페이스(`Port`)를 호출합니다.
- **철저한 흐름 통제(Orchestration):** 서비스는 스스로 헤비한 비즈니스 규칙을 가지지 않습니다. 대신 정문으로 들어온 배달통(`Command`)을 열고, 도메인 객체(`Money`, `Transaction`)들을 깨워 비즈니스를 수행하게 한 뒤, 그 결과를 뒷문(`Port`)에 위임하는 흐름 제어의 역할만 수행합니다.

---

## 2. 핵심 제어 원칙 (Core Implementation Rules)

외부 기술의 변화에 코어 비즈니스가 흔들리지 않도록 서비스 구현 시 아래의 3대 제약 조건을 엄격히 준수해야 합니다.

### ① 스프링 프레임워크 어노테이션의 완벽한 배제
- 클래스 상단에 `@Service`, `@Component`, `@Autowired` 같은 스프링 프레임워크 전용 어노테이션을 **절대 사용하지 않습니다.**
- **이유:** 이 모듈은 순수한 코틀린 객체(POJO) 영역이어야 합니다. 특정 프레임워크 기술에 종속되는 순간, 프레임워크 버전 업그레이드나 기술 전환 시 비즈니스 코어까지 오염됩니다.
- **대안:** 빈(Bean) 등록 및 의존성 주입은 가장 바깥쪽의 설정 모듈(`:bootstrap` 또는 `:config`)에서 수동 자바/코틀린 설정을 통해 처리합니다.

### ② 데이터베이스 트랜잭션(@Transactional) 격리
- 서비스 클래스나 메서드에 `@Transactional` 어노테이션을 **사용하지 않습니다.**
- **이유:** 트랜잭션은 영속성 계층(DB, JPA)의 하부 기술적 개념입니다. 순수 비즈니스 서비스는 데이터가 '어떻게 동기화되고 롤백되는지' 알 필요가 없습니다.
- **대안:** 트랜잭션 경계 설정은 인프라 웹 어댑터나 별도의 트랜잭션 데코레이터/인터셉터 계층에서 관장하도록 설계합니다.

### ③ 생성자 주입(Constructor Injection) 강제 및 불변성 유지
- 필드 주입은 일절 금지하며, 서비스가 필요로 하는 모든 아웃바운드 포트는 **클래스 기본 생성자**를 통해 주입받아야 합니다.
- 주입받는 모든 포트 필드는 반드시 `private val`로 선언하여 한 번 주입된 포트가 런타임에 변경될 수 없도록 불변성을 강제합니다.

## 3. 데이터의 흐름 및 표준화 (Data Flow Spec)

외부 개발자가 가계부 등록 유즈케이스(`RecordTransactionService`)를 이해하기 위한 데이터 변환 및 흐름 명세는 다음과 같습니다.

```text
[외부 날것의 데이터 (Double, String)] 
       │
       ▼ (1) Web Controller가 수신 후 Command 객체로 매핑 (1차 Syntax 검증 완료)
[RecordTransactionCommand]
       │
       ▼ (2) Service의 record() 메서드 호출 및 진입
   ┌───┴───────────────────────────────────────────┐
   │ 1. 자산 데이터 표준화                             │
   │    command.amount (Double)                    │
   │    ──> Money.of() ──> [Money (BigDecimal)]     │
   │                                               │
   │ 2. 도메인 엔티티 조립                           │
   │    개별 파라미터 ──> [Transaction Entity]       │
   │    (내부적으로 식별자 'TX-UUID' 규칙 자동 발급)  │
   └───┬───────────────────────────────────────────┘
       │
       ▼ (3) 완성된 도메인 객체를 뒷문으로 위임 (saveTransactionPort.save(transaction))
[SaveTransactionPort]
